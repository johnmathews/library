# Conversational Ask

Sub-project 4 of the extraction/Ask roadmap. Makes Ask multi-turn: a follow-up
like *"what about last year?"* resolves against the prior turns of the same
conversation rather than being answered cold.

## Five locked decisions

1. **State lives server-side** in two new tables (`ask_threads`, `ask_turns`).
   Conversations are durable and resumable by id; no client-side state required.

2. **Follow-ups re-feed prior Q&A *and* prior tool results** into the Claude
   loop. The model can reason over earlier evidence without re-querying. Cost is
   bounded by decision 3.

3. **Sliding turn window + prompt caching.** Re-feed the last
   `LIBRARY_ASK_HISTORY_TURNS` turns (default 3) verbatim; drop older turns.
   The rehydrated history prefix and the static system+tools definitions each
   carry an Anthropic `cache_control: ephemeral` breakpoint so re-sent turns hit
   the prompt cache. Once a thread exceeds the window, the history-prefix cache
   misses on dropped turns (the cache key changes); system+tools remain cached.
   Accepted — most threads are short.

4. **Chat UI + conversation sidebar.** `AskView` is a chat transcript with a
   follow-up input. A sidebar lists past threads by title and relative time;
   clicking resumes. Routes: `/ask` (new) and `/ask/:threadId` (resume).

5. **`ask_logs` dropped; `ask_turns` subsumes it.** `ask_logs` was write-only
   (no readers anywhere in the codebase). `ask_turns` is the single source of
   cost/provenance. Existing rows discarded — acceptable on a young, write-only
   table. Migration 0008 drops `ask_logs` and creates `ask_threads` + `ask_turns`.

## `ask_turns.messages` replay-blob approach

Replaying prior tool results faithfully means handing the Anthropic API the
exact `tool_use`/`tool_result` block sequence again. Rather than reconstruct it
from `citations`/`used_tools`, each turn serializes the Anthropic message blocks
it produced into `ask_turns.messages` (JSONB). The serialized form is plain
dicts (`block.model_dump(mode="json")`) — valid Anthropic API input — so replay
is just prepending the flattened blobs as the history prefix. The `query` and
`answer` columns are denormalized convenience fields for display; the question
also appears as the first block in `messages` (small deliberate duplication,
keeps replay robust).

## Prompt-caching trade-off

Caching is best-effort: correctness never depends on a cache hit. When a thread
exceeds the `LIBRARY_ASK_HISTORY_TURNS` window, the oldest turns are dropped.
This changes the start of the history prefix, causing the cache to miss for that
re-send. The static system prompt and tool definitions (marked with their own
breakpoint) remain cached regardless. For typical short conversational threads
this is a good trade-off: bounded token cost, low implementation complexity,
predictable cache behaviour.

## SDD execution

8 tasks generated by the SDD harness from the design spec
(`docs/superpowers/specs/2026-06-22-conversational-ask-design.md`):

1. Data model — drop `ask_logs`, add `ask_threads` + `ask_turns` (migration 0008)
2. Engine — `run_ask` gains `history_messages` param and returns `turn_messages`
3. API — `POST /api/ask` threading + thread CRUD endpoints
4. API tests — coverage of new/continue thread paths, ownership, 404s
5. Frontend API client — `askQuestion`, `listThreads`, `getThread`, `deleteThread`
6. Frontend chat transcript component
7. Frontend conversation sidebar + routing
8. Docs + journal (this entry)

A code-review agent ran after tasks 1–7 and found several issues: missing
`thread.updated_at` bump on the first turn (fixed inline), a stale test helper
(`_ask_logs_count` → `_ask_turns_count`), and minor type/lint findings. All
were resolved before the final commit.

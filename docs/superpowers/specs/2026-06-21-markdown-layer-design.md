# Document markdown layer + page-aware citations

**Status:** approved design, pre-implementation (2026-06-21).
**Sub-project:** 2 (and 3) of 5 in the extraction/Ask improvement track. Sub-project 1
(extraction quality — validation, review queue, eval harness) is merged. This spec folds
the originally-separate sub-project 3 (PDF page-number citations) into sub-project 2,
because vision generates markdown per page and page provenance is then nearly free — and
both touch the chunk model, which is better changed once. The remaining items
(conversational Ask; document series + comparative queries) get their own specs later.

## Problem

The pipeline's only text representation of a document is `ocr_text`: a flat blob whose
layout has been discarded (PDF pages are joined with blank lines, tables collapse into
runs of words). This hurts three things:

- **Chunking/embeddings** are built from layout-less text, so table rows, key/value
  pairs, and borderless tables embed as word soup — weaker semantic retrieval.
- **Ask context** is the same word soup; the model reconstructs structure on the fly.
- **Citations** can only point at a document and an opaque chunk ordinal, never a page —
  `chunk_index` is a position in a page-less stream (see [ask.md](../../ask.md) §1.6.1).

## Goal

Produce a clean, canonical **markdown rendering** of each document — real markdown
tables (including borderless tables reconstructed from the page image), headings, and
lists — generated by Claude **vision** grounded on the OCR text. Store it **per page** so
that:

1. **Chunking/embeddings** are built from the structured markdown, improving semantic
   retrieval;
2. **Ask context** and the document detail view can show the "understood" representation;
3. **Citations carry a page number** and the detail view can deep-link the PDF to that
   page (the folded-in sub-project 3).

## Decisions (from brainstorming)

- **Generation = vision + OCR-text hybrid.** Render each page to an image and send all
  page images **plus the full `ocr_text` as a grounding aid** in one structured-output
  call. Vision recovers layout/tables; the OCR text anchors exact numbers and names so
  the model hallucinates less. (Rejected: OCR-text reformatting — cannot reconstruct
  borderless tables; vision-only — more prone to misreading exact digits.)
- **Fold sub-project 3 in fully.** Deliver the markdown layer, page-aware chunking, **and**
  the user-facing page citations (Ask → API → frontend deep-link) in one pass, touching
  the chunk model exactly once.
- **Storage = `document_pages` table.** Normalized per-page markdown as the canonical
  source of truth; `document_chunks` gains `page_number`. (Rejected: flat column with
  page-break sentinels — fragile parsing; JSONB in `extra` — large blob, awkward to query,
  `extra` already crowded.)
- **New pipeline stage `markdown`** between `extract` and `embed`; embed chunks from the
  per-page markdown. Best-effort, exactly like extraction/embed.
- **FTS unchanged** — stays on `ocr_text` (exact OCR terms); markdown feeds embeddings,
  Ask, and display only.
- **Default model = `claude-haiku-4-5`** (household tables are simple; cheap). No
  escalation this phase.
- **Own daily budget guard** (`LIBRARY_MARKDOWN_DAILY_BUDGET_USD`), separate from the
  extraction budget, for independent control.

## Architecture

### 1. Generation — `src/library/markdown/`

Module mirrors the shape of `src/library/extraction/` (a `generator.py` doing the call, a
`schema.py` Pydantic model, an `apply.py` writing rows). `PROMPT_VERSION` is recorded with
every run, as extraction does, so a prompt change can identify stale renderings.

**Rendering.** Pages are rasterized with pypdfium2 (already used by thumbnails, the OCR
confidence probe, and the photo-path gate). Each page becomes one image, downscaled so the
long side is at most `LIBRARY_MARKDOWN_IMAGE_LONG_SIDE_PX` (default 1600) — enough for
table fidelity, bounded token cost. Input-type handling:

| Input | Pages rendered |
|---|---|
| `application/pdf` | each page (capped at `MARKDOWN_MAX_PAGES`) |
| `image/tiff` | the OCR-produced PDF's pages (TIFF is converted to PDF in OCR) |
| `image/jpeg`, `image/png` | the single image → one page |
| `image/heic`, `image/heif` | the derived `converted.jpg` → one page |
| `text/plain` | **skipped** — no layout to recover; embed falls back to `ocr_text` |

**Call.** One `client.messages.parse()` call (async SDK, structured outputs — the same
pattern as `extraction/schema.py`) with content `[page images…] + [full ocr_text as
grounding] + [instruction]` and schema `list[PageMarkdown]` where
`PageMarkdown = {page_number: int, markdown: str}`. Structured output removes
delimiter-parsing fragility and lets us validate page coverage against the image count.
`max_tokens` is set generously (full-document markdown can be large).

**Long documents.** Render at most `LIBRARY_MARKDOWN_MAX_PAGES` (default 20). Pages are
sent in batches of `LIBRARY_MARKDOWN_PAGE_BATCH` (default 10) with a page-number offset per
batch; the per-batch results are concatenated. A document over the page cap renders only
its first `MARKDOWN_MAX_PAGES` pages (logged); the remaining pages have no markdown and
their chunks fall back to `ocr_text` page-less chunking.

**Coverage validation.** Returned `page_number`s are validated against the rendered image
count for the batch. A page the model returns no markdown for gets **no row** (it simply
yields no chunks); duplicate or out-of-range page numbers are dropped with a warning. A
wholly unparseable response is treated as a generation failure (best-effort skip), never a
document failure.

### 2. Storage (migration 0007)

- **`document_pages`** — new table:

  | column | type | notes |
  |---|---|---|
  | `document_id` | FK `documents.id` ON DELETE CASCADE | |
  | `page_number` | `int` | 1-based |
  | `markdown` | `text` | the page's markdown |
  | `char_count` | `int` | length of `markdown` (cheap diagnostics) |
  | `created_at` | `timestamptz` | |

  Primary key `(document_id, page_number)`. The relationship on `Document` uses
  `lazy="raise"` + `passive_deletes=True`, mirroring `chunks` (per-page markdown is never
  wanted on a normal document load). Full-document markdown = pages ordered by
  `page_number`, assembled on demand.

- **`document_chunks.page_number`** — new `int | None` column. `NULL` means "no page
  provenance" (document had no markdown — text-only, skipped, or failed; or a page past the
  render cap).

- No backfill in the migration (fast, reversible) — see Backfill.

### 3. Pipeline integration

New status in the lifecycle: **`received → ocr → extract → markdown → embed → indexed`**
(add `DocumentStatus.MARKDOWN`). Stage hook `async def run_markdown(session, document)` in
`library.jobs`, placed after `run_extraction` and before `run_embed`:

- Entering `markdown` runs generation and writes `document_pages` (replacing any existing
  rows for the document — idempotent re-run). It records an ingestion event and continues.
- **Best-effort**, identical contract to extraction: `LIBRARY_MARKDOWN_ENABLED=false`,
  missing API key, over budget, unusable/oversized input, or an API error after SDK
  retries ⇒ the document still proceeds to `embed` and reaches `indexed`. Only the pipeline
  machinery itself can fail a document.

The **embed** stage (`run_embed`) changes to be page-aware:

- If `document_pages` exist: chunk **each page's markdown** with the existing
  `chunk_text` packer, carrying `chunk_index` continuously across pages and tagging each
  `DocumentChunk` with its `page_number`.
- Else (no pages): chunk `ocr_text` as today, `page_number = NULL`.

CPU-bound rasterization runs in a thread (`asyncio.to_thread`), like OCR.

### 4. Ask — page-numbered citations

- `semantic_search` already returns each hit's nearest chunk; expose that chunk's
  `page_number` on the hit. In `ask.engine`, track a `document_id → page_number` map from
  the top text hit per document (alongside the existing `cited` set).
- `AskCitation` gains `page_number: int | None`. `_citations_for` fills it from the map;
  aggregation citations (`query_documents`) have no text location, so `page_number` stays
  `None`.
- `/api/ask` response schema carries `page_number` per citation.

### 5. Frontend

- **`AskView.vue`** — render each citation as `Title, p.N` (omit `, p.N` when `page_number`
  is `None`), linking to the document detail view at that page.
- **`DocumentDetailView.vue`** — deep-link the PDF iframe to the cited page via the URL
  fragment: `#page=N` combined with the existing `#toolbar=0&navpanes=0&view=FitH`. Image
  documents are always page 1.
- **Markdown tab** in the detail view rendering the assembled document markdown (the
  "understood" representation), behind a new read API (below).

### 6. API additions

- `GET /api/documents/{id}/markdown` — returns the assembled per-page markdown (ordered),
  e.g. `{pages: [{page_number, markdown}], page_count}`; 404/empty when none. Backs the
  detail-view markdown tab.
- `page_number` added to the `/api/ask` citation schema (above).

### 7. Backfill

`library backfill-markdown [--limit N]` — for documents lacking `document_pages`, generate
them and re-embed page-aware. Idempotent (replaces a document's pages and chunks), mirrors
`backfill-embeddings` / `backfill-validation`. The worker must be running. On CPU the
re-embed is slow for a large corpus; throttle with `--limit`.

### 8. Configuration (new settings, `LIBRARY_` prefix)

| Setting | Default | Purpose |
|---|---|---|
| `LIBRARY_MARKDOWN_ENABLED` | `true` | Master switch for the markdown stage. |
| `LIBRARY_MARKDOWN_MODEL` | `claude-haiku-4-5` | Vision model for markdown generation. |
| `LIBRARY_MARKDOWN_DAILY_BUDGET_USD` | `5.0` | Daily spend cap for markdown generation (own guard, mirrors extraction's; over budget ⇒ skip with `reason: "budget"`). |
| `LIBRARY_MARKDOWN_MAX_PAGES` | `20` | Max pages rendered/sent per document. |
| `LIBRARY_MARKDOWN_PAGE_BATCH` | `10` | Pages per vision call (batched with page offset). |
| `LIBRARY_MARKDOWN_IMAGE_LONG_SIDE_PX` | `1600` | Long-side cap for rendered page images. |

Cost is recorded per run in the `markdown_completed` ingestion event (`model`,
`prompt_version`, `input_tokens`, `output_tokens`, `cost_usd`, `pages`); the daily budget
sums today's `cost_usd` across markdown events, exactly like the extraction guard.

### 9. Ingestion events

| event | detail |
|---|---|
| `markdown_completed` | `{model, prompt_version, pages, input_tokens, output_tokens, cost_usd}` |
| `markdown_skipped` | `{reason, …}` — `disabled`, `missing_api_key`, `budget`, `input_unusable`, `file_too_large` |
| `markdown_failed` | `{error, prompt_version}` |

## Testing

- **Unit**: page-coverage validation (missing/duplicate/extra page numbers), page-batch
  offsetting, page-aware chunking + `page_number` assignment (incl. the `ocr_text`
  fallback ⇒ `NULL`), budget guard math, skip conditions (`text/plain`, disabled, no key,
  over cap).
- **Integration**: `run_markdown` writes `document_pages` (vision call mocked) and the
  best-effort fallbacks leave the document reaching `indexed`; `run_embed` produces
  page-tagged chunks from pages and page-less chunks from `ocr_text`; `backfill-markdown`
  over a seeded corpus; `GET /api/documents/{id}/markdown`.
- **Ask**: a citation carries `page_number` from the top semantic hit; aggregation
  citations carry `None`; `/api/ask` schema includes it.
- **Frontend** (Vitest + @vue/test-utils): citation renders `Title, p.N` and omits the
  page when `None`; the detail-view deep-link URL includes `#page=N`; the markdown tab
  renders pages. **One Playwright e2e**: ask a question → citation shows a page → click
  deep-links the PDF preview to that page.

## Non-goals (this phase)

Markdown feeding the *extraction* input (extraction stays on `ocr_text`); cross-encoder
re-ranking of retrieval; a markdown editing/correction UI; automatic re-render on prompt
change beyond the `backfill-markdown` CLI; per-page OCR confidence; markdown in FTS;
escalation to a stronger model on low-quality markdown.
